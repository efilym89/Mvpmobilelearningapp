# Annaelle Learning MVP

Репозиторий MVP-продукта для обучения персонала. Приложение должно помогать сотруднику изучать материалы, проходить экзамены и позволять администратору управлять курсами, тестами и результатами.

## Источники решений

Новые изменения в проекте теперь должны опираться на эти документы:

- [idea.md](idea.md) — продуктовая идея и базовый пользовательский сценарий.
- [vision.md](vision.md) — техническое видение MVP, целевая архитектура и границы решения.
- [conventions.md](conventions.md) — правила разработки и выбора решений.
- [guidelines/Guidelines.md](guidelines/Guidelines.md) — короткая рабочая памятка для команды и code-ассистента внутри этого репозитория.

Если между текущим кодом и новыми решениями возникает конфликт, приоритет у `vision.md`, затем у `conventions.md`.

## Текущее состояние

Сейчас в репозитории уже есть рабочий `Vite/React`-прототип в каталоге `src/`. Он покрывает ключевые пользовательские сценарии MVP:

- просмотр доступных курсов;
- изучение уроков;
- прохождение тестов;
- просмотр прогресса;
- базовый административный экран.

Этот прототип сохраняется как быстрый слой проверки UX и сценариев. При этом дальнейшее развитие проекта теперь должно идти в сторону целевой структуры из `vision.md`.

## Целевая структура проекта

Проект начинает переход к `pnpm`-monorepo с такой логикой:

- `apps/mobile` — мобильное приложение для сотрудников;
- `apps/admin` — административная панель;
- `apps/backend` — единый backend с `REST API`;
- `packages/shared-contracts` — общие типы, DTO и контракты;
- `packages/tooling` — общие настройки инструментов;
- `docs` — документация проекта;
- `infra` — минимальная инфраструктурная обвязка.

До полной миграции текущий каталог `src/` остаётся временным web-прототипом MVP.

## MVP-рамки

В проект внедрены следующие опорные ограничения:

- делаем только то, что нужно для проверки идеи;
- выбираем самое простое рабочее решение по `KISS`;
- не добавляем микросервисы, `CQRS`, event-driven архитектуру и enterprise-усложнения;
- бизнес-логика должна жить на backend, а не разрастаться во frontend;
- общие контракты выносятся только при реальном повторном использовании;
- конфигурация должна идти через переменные окружения, без секретов в репозитории.

## Репозиторий

- `src/` — текущий интерактивный прототип MVP;
- `src/domain/` — доменные типы и константы, закрепляющие новые MVP-правила;
- `apps/`, `packages/`, `docs/`, `infra/` — каркас под целевую структуру;
- `guidelines/` — внутренние правила работы с репозиторием.

## Запуск текущего прототипа

Пока основной исполняемый код находится в `src/`:

```bash
npm install
npm run dev
```

## Запуск Preview API

Чтобы frontend работал не только на `mock-data`, в проект добавлен локальный preview backend.

```bash
npm run dev:api-preview
```

По умолчанию API поднимается на `http://localhost:4100/api`.

Демо-доступы:

- `employee@annaelle.ru` / `demo123`
- `admin@annaelle.ru` / `demo123`

## Ближайшие шаги

1. Перенести прототип в целевую monorepo-структуру без потери текущих сценариев MVP.
2. Выделить общие API-контракты для mobile, admin и backend.
3. Заменить mock-данные на backend-ориентированную модель в рамках `vision.md`.
