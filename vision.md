# Техническое видение проекта

## 1. Цель документа и границы MVP

Документ фиксирует техническое видение MVP приложения для обучения персонала. Он служит отправной точкой для последующей разработки и помогает удерживать проект в рамках простого и практичного решения.

Базовый принцип документа: реализуем только то, что необходимо для проверки идеи. Любые решения, которые усложняют разработку без прямой пользы для MVP, откладываются.

На этапе MVP система должна решать только ключевые задачи:
- предоставить сотруднику учебные материалы в мобильном приложении;
- дать сотруднику возможность пройти экзамен по материалам;
- предоставить простой административный интерфейс для управления учебными материалами и экзаменами;
- хранить результаты прохождения обучения и экзаменов.

## 2. Технологии

Для MVP выбираем следующий минимально необходимый стек:

- мобильное приложение: `Flutter` для единой кодовой базы под `iOS` и `Android`;
- backend: `NestJS` как единый сервер приложения;
- API: `REST`;
- база данных: `PostgreSQL`;
- административная панель: `Next.js + TypeScript`;
- репозиторий: `pnpm`-monorepo.

Технологические ограничения MVP:

- без микросервисов;
- без отдельного `Redis` на старте;
- без сложной офлайн-синхронизации;
- без `SSO` и сложных интеграций на первом этапе.

## 3. Принципы разработки

На этапе MVP проект разрабатывается по принципам простоты и предсказуемости. Главная цель — как можно быстрее собрать рабочее решение для проверки идеи, не усложняя архитектуру заранее.

Базовые принципы разработки:

- `KISS`: выбираем самое простое решение, которое закрывает бизнес-задачу;
- один монорепозиторий для всех частей системы;
- один backend как модульный монолит;
- одно мобильное приложение на `Flutter` для `iOS` и `Android`;
- `REST API` как основной способ взаимодействия клиента и сервера;
- сначала реализуем базовый рабочий сценарий, затем улучшаем;
- не используем сложные архитектурные подходы без прямой необходимости;
- не вводим дополнительные сервисы и инфраструктуру, если можно обойтись без них;
- покрываем тестами только критически важные сценарии `MVP`.

На этапе MVP намеренно не используем:

- микросервисы;
- `CQRS`;
- event-driven архитектуру;
- сложную систему интеграций;
- избыточные уровни абстракции в коде.

## 4. Структура проекта

Проект организуется как `monorepo`, чтобы мобильное приложение, backend, административная панель и общие контракты находились в одном месте. Это упрощает разработку, навигацию по проекту и поддержку MVP.

Предлагаемая структура проекта:

- `apps/mobile` — мобильное приложение для сотрудников;
- `apps/admin` — административная панель для управления учебными материалами и экзаменами;
- `apps/backend` — единый backend с `REST API`;
- `packages/shared-contracts` — общие типы, DTO и контракты API;
- `packages/tooling` — общие настройки линтера, форматирования и базовых конфигураций;
- `docs/` — документация проекта;
- `infra/` — минимальная инфраструктурная обвязка для локального запуска и деплоя.

Принципы структуры:

- каждый каталог отвечает за одну понятную задачу;
- не создаем дополнительные пакеты заранее;
- общие сущности выносим только тогда, когда они реально используются в нескольких приложениях;
- структура должна быть понятной без дополнительного объяснения.

## 5. Архитектура проекта

Архитектура проекта для MVP строится вокруг одного backend-приложения, которое обслуживает мобильное приложение сотрудников и административную панель. Основная цель такой архитектуры — сократить количество компонентов и упростить разработку.

Архитектурная схема MVP:

- мобильное приложение и административная панель работают только через `REST API`;
- backend является единой точкой бизнес-логики;
- backend отвечает за аутентификацию, выдачу учебных материалов, прием экзаменов и сохранение результатов;
- все данные хранятся в одной базе `PostgreSQL`;
- при необходимости файлы учебных материалов могут храниться отдельно, но на старте можно ограничиться хранением ссылок и текстового контента в базе.

Модули backend на этапе MVP:

- `auth` — вход пользователя в систему;
- `users` — сотрудники и администраторы;
- `courses` — учебные курсы;
- `lessons` — уроки и материалы;
- `tests` — экзамены, вопросы и варианты ответов;
- `results` — результаты экзаменов и прогресс обучения.

Архитектурные принципы:

- один сервер вместо набора сервисов;
- одна база данных вместо распределенного хранения;
- бизнес-логика находится на backend, а не в клиентах;
- клиенты отображают данные и вызывают API;
- каждая функциональная область оформляется как отдельный модуль, но все модули живут в одном приложении.

## 6. Модель данных

Модель данных для MVP должна покрывать только основные сущности, необходимые для обучения и экзаменов. Мы не проектируем универсальную LMS, а фиксируем минимальный набор объектов.

Основные сущности:

- `User` — пользователь системы;
- `Role` — роль пользователя;
- `Course` — учебный курс;
- `Lesson` — урок внутри курса;
- `LessonContent` — содержимое урока;
- `Test` — экзамен или тест;
- `Question` — вопрос теста;
- `AnswerOption` — вариант ответа;
- `TestAttempt` — попытка прохождения теста;
- `TestAnswer` — ответ пользователя в рамках попытки;
- `CourseProgress` — прогресс пользователя по курсу.

Минимальные связи:

- пользователь имеет одну или несколько ролей;
- курс содержит несколько уроков;
- урок содержит одно или несколько содержимых элементов;
- тест относится к курсу или конкретному уроку;
- тест содержит несколько вопросов;
- вопрос содержит несколько вариантов ответа;
- пользователь может иметь несколько попыток прохождения теста;
- прогресс пользователя хранится отдельно от структуры курса.

Минимальный состав ролей:

- `employee` — изучает материалы и проходит экзамены;
- `admin` — загружает материалы, создает тесты и просматривает результаты.

Упрощения для MVP:

- без сложной оргструктуры;
- без версионирования курсов;
- без отдельной таблицы назначений по подразделениям на старте;
- без сложной системы прав доступа;
- без аналитического хранилища и отдельных проекций.

## 7. Сценарии работы

На этапе MVP поддерживаются только ключевые сценарии, позволяющие проверить ценность продукта.

Основные пользовательские сценарии:

- администратор входит в административную панель;
- администратор создает курс;
- администратор добавляет в курс уроки и учебные материалы;
- администратор создает тест и вопросы к нему;
- администратор назначает курс сотрудникам или делает его доступным целевой группе простым способом;
- сотрудник входит в мобильное приложение;
- сотрудник видит список доступных курсов;
- сотрудник открывает курс и изучает материалы;
- сотрудник проходит тест;
- система сохраняет результат теста и обновляет прогресс;
- администратор просматривает результаты прохождения.

Сценарии, которые можно отложить:

- push-уведомления;
- офлайн-режим с синхронизацией;
- продвинутая аналитика;
- повторные сложные траектории обучения;
- интеграции с внешними HR-системами.

## 8. Деплой

Подход к деплою для MVP должен быть максимально простым и воспроизводимым.

Базовый подход:

- backend разворачивается как одно приложение;
- `PostgreSQL` разворачивается как отдельный сервис;
- административная панель разворачивается отдельно;
- мобильное приложение собирается и публикуется как обычное приложение для `iOS` и `Android`;
- для локальной разработки используем `Docker Compose`.

Минимальные среды:

- `local` — локальная разработка;
- `dev` — тестовый стенд для проверки командой;
- `prod` — рабочая среда.

Принципы деплоя:

- одинаковый способ запуска на всех средах;
- минимум ручных действий;
- конфигурация через переменные окружения;
- без `Kubernetes` и сложной оркестрации на старте.

## 9. Подход к конфигурированию

Конфигурирование проекта строится на простом и понятном подходе через переменные окружения.

Основные правила:

- все настройки среды задаются через `.env`-файлы и переменные окружения;
- значения по умолчанию допустимы только для локальной разработки;
- секреты не хранятся в репозитории;
- backend, admin и mobile используют только свой необходимый набор настроек.

Минимальный набор конфигурации:

- адрес backend API;
- параметры подключения к `PostgreSQL`;
- секреты для авторизации;
- адрес административной панели;
- флаги окружения `local`, `dev`, `prod`.

Принципы:

- конфигурация должна быть плоской и читаемой;
- не вводим собственную сложную систему конфигов;
- добавляем новые параметры только по мере реальной необходимости.

## 10. Подход к логгированию и мониторингу

На этапе MVP логгирование должно помогать разработке и поддержке, но не превращаться в отдельный большой контур.

Базовый подход к логгированию:

- backend пишет структурированные логи основных событий;
- ошибки API логируются обязательно;
- успешные и неуспешные попытки входа логируются;
- создание и изменение учебных материалов со стороны администратора логируется;
- прохождение тестов и сохранение результатов логируется на базовом уровне.

Что логируем обязательно:

- время события;
- тип события;
- идентификатор пользователя;
- идентификатор запроса или `traceId`;
- результат выполнения операции;
- текст ошибки при сбое.

Мониторинг на старте:

- мониторим доступность backend;
- мониторим ошибки приложения;
- мониторим базовые технические метрики, если это легко подключается;
- без отдельной сложной observability-платформы на старте.

Принцип:

- сначала понятные логи и базовая диагностика;
- затем, при необходимости, расширяем мониторинг.

## 11. Что намеренно не делаем на этапе MVP

Чтобы сохранить скорость разработки и не усложнять систему раньше времени, на этапе MVP сознательно не реализуем следующие возможности:

- микросервисную архитектуру;
- сложную ролевую модель и тонкую настройку прав;
- интеграции с корпоративными системами;
- `SSO` и внешние провайдеры авторизации;
- офлайн-режим с полной синхронизацией;
- push-уведомления и отдельный сервис уведомлений;
- расширенную аналитику и дашборды;
- версионирование учебного контента;
- конструктор сложных учебных траекторий;
- файловое хранилище и медиапайплайн корпоративного уровня;
- отказоустойчивую распределенную инфраструктуру;
- сложный аудит всех действий на уровне enterprise.

Если какая-либо из этих возможностей подтвердится как действительно необходимая после проверки идеи, она должна добавляться отдельным следующим этапом, а не заранее.
