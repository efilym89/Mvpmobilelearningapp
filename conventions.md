# Соглашение по разработке

## Назначение

Этот документ задает краткие правила для code-ассистента, который генерирует код в проекте. Базовый источник решений — [vision.md](vision.md). Если возникает спор между новым решением и текущим техническим видением, приоритет у `vision.md`.

## Главные правила

1. Пиши код только в рамках `MVP` и текущих сценариев из `vision.md`.
2. Всегда выбирай самое простое рабочее решение по принципу `KISS`.
3. Не добавляй архитектуру, сервисы, библиотеки и абстракции, которых нет в `vision.md`, без явной необходимости.
4. Сохраняй текущую структуру `monorepo` и размещай код только в подходящем приложении или пакете.
5. Соблюдай модульность: один модуль отвечает за одну понятную бизнес-задачу.
6. Не переноси бизнес-логику во frontend, если она должна жить на backend.
7. Используй только согласованный технологический стек из `vision.md`.
8. Не внедряй заранее исключенные подходы из `vision.md`: никакого оверинжиниринга, преждевременной универсализации и enterprise-усложнений.
9. Общие типы и контракты выноси в shared-пакеты только при реальном повторном использовании.
10. Конфигурацию храни через переменные окружения; не хардкодь секреты, URL и параметры среды.
11. Логи добавляй только там, где они помогают диагностике ключевых сценариев и ошибок.
12. Тесты пиши только для критичных сценариев `MVP`, не раздувая тестовую инфраструктуру.

## Правило выбора

Если есть несколько вариантов реализации, всегда выбирай тот, который:

- требует меньше кода;
- проще поддерживается;
- лучше соответствует `vision.md`;
- быстрее приводит к рабочему результату.
