Обнови и доработай текущий дизайн мобильного приложения обучения персонала, НЕ меняя согласованный брендовый визуальный стиль.

Важно:
- Сохрани текущую визуальную систему, стилистику, настроение, композицию, цветовую палитру, типографику, скругления, паттерны карточек, иконографику и общую эстетику, так как дизайн уже согласован по брендбуку.
- Не делай редизайн. Делай только продуктовые и UX-улучшения внутри существующего визуального языка.
- Интерфейс для конечного пользователя должен остаться на русском языке.
- Для удобства дальнейшей разработки названия фреймов, компонентов и слоев делай на английском, но весь пользовательский текст оставляй на русском.
- Макет должен быть подготовлен так, чтобы по нему было удобно проектировать логику, backend, API, роли, состояния, сущности и пользовательские сценарии.

Контекст продукта:
Это внутреннее мобильное LMS-приложение для обучения сотрудников.
Основной сценарий:
1. пользователь получает доступ к системе
2. видит назначенные курсы
3. открывает курс
4. проходит уроки
5. после уроков проходит квизы
6. после завершения курса проходит итоговый тест
7. получает результат
8. при успехе получает сертификат

Роли:
1. Employee
- обычный пользователь
- проходит обучение
- НЕ имеет доступа к админке

2. Admin
- пользователь с выданными правами
- кроме обычного сценария обучения имеет доступ к дополнительной странице администрирования
- эта страница должна называться "Дашборд"

Ключевые изменения, которые нужно внести в дизайн:

1. Навигация
- Объедини вкладки "Прогресс" и "Профиль" в одну общую вкладку "Профиль"
- В новой вкладке "Профиль" красиво и логично совмести:
  - личные данные пользователя
  - статистику обучения
  - прогресс по курсам
  - достижения
  - сертификаты
- Для обычного пользователя должно быть 3 нижние вкладки:
  - Главная
  - Курсы
  - Профиль
- Для пользователя с правами Admin должно быть 4 нижние вкладки:
  - Главная
  - Курсы
  - Профиль
  - Дашборд

2. Админка / Dashboard
- Добавь отдельную страницу "Дашборд" только для пользователей с правами доступа
- Вкладка "Дашборд" не должна отображаться у обычных пользователей
- Продумай состояние, если пользователь без прав попадает в Dashboard по прямой ссылке: аккуратный экран "Нет доступа" в фирменном стиле
- Dashboard должен быть мобильным, в том же брендовом стиле, без ощущения отдельного продукта
- Dashboard должен включать минимально необходимые блоки для MVP:
  - список курсов
  - создание/редактирование курса
  - список уроков внутри курса
  - создание/редактирование тестов
  - просмотр результатов сотрудников
  - управление назначениями курсов
- Не превращай Dashboard в перегруженную enterprise-систему. Это должен быть чистый, понятный, мобильный admin MVP.

3. Улучшение продуктовой логики в UI
Дизайн должен явно поддерживать и визуально объяснять:
- назначенные пользователю курсы
- обязательные курсы
- рекомендованные курсы
- статус курса:
  - не начат
  - в процессе
  - завершен
  - просрочен
- дедлайн курса
- прогресс курса
- статус урока:
  - доступен
  - заблокирован
  - просмотрен
  - завершен
- статус квиза:
  - не начат
  - пройден
  - не пройден
  - доступен для повтора
- статус итогового теста:
  - доступен
  - в процессе
  - сдан
  - не сдан
  - доступен для пересдачи

4. Course UX
Доработай карточку курса и экран курса так, чтобы было понятно:
- курс назначен пользователю или просто доступен в каталоге
- обязательный ли курс
- есть ли дедлайн
- сколько уроков завершено
- какой следующий шаг
- что именно нужно сделать для завершения курса
- когда откроется следующий урок
- итоговый тест доступен только после выполнения условий

5. Lesson UX
Доработай экран урока:
- визуально покажи, что завершение урока зависит не только от открытия видео, а от выполнения условия завершения
- добавь понятный блок "Что нужно для завершения урока"
- добавь состояние:
  - урок просмотрен частично
  - урок завершен
  - квиз обязателен для перехода дальше
- сделай понятный CTA:
  - продолжить просмотр
  - пройти квиз
  - пересмотреть урок
  - вернуться к курсу

6. Quiz and Test UX
Добавь полноценные экраны и состояния для:
- стартового экрана квиза
- стартового экрана итогового теста
- информации перед началом:
  - количество вопросов
  - проходной балл
  - лимит попыток
  - таймер, если он нужен
- процесса прохождения
- подтверждения завершения
- результата:
  - успешно
  - неуспешно
  - сколько правильных ответов
  - что делать дальше
- пересдачи
- недоступности теста, если курс не завершен

7. Profile tab
Сделай объединенный экран "Профиль" как умный хаб пользователя:
- блок профиля
- статистика обучения
- текущий уровень / достижения
- завершенные курсы
- сертификаты
- средний балл
- часы обучения
- текущие курсы
- при необходимости быстрые действия:
  - выйти
  - поддержка
  - настройки
- экран должен быть логичным, а не просто склейкой старых экранов "Прогресс" и "Профиль"

8. Системные состояния
Подготовь дизайн не только happy path, но и для реальной разработки:
- loading
- empty state
- no assigned courses
- no search results
- no access
- network error
- expired session
- test unavailable
- certificate unavailable
- dashboard unavailable for user without rights

9. Auth and access
Сохрани текущий стиль экранов входа, но уточни UX:
- login
- request access
- success state after request access
- invalid credentials
- forgot password
Нужно, чтобы из дизайна было понятно:
- обычный пользователь входит по выданному доступу
- пользователь без доступа может оставить заявку
- доступ к Dashboard определяется правами пользователя

10. Требования к структуре макета для handoff в разработку
Подготовь макет максимально чисто и системно:
- screen-by-screen flow
- отдельные экраны для Employee и Admin where needed
- единые компоненты и variants
- явные названия состояний
- reusable components for:
  - course card
  - lesson item
  - quiz option
  - result card
  - status badge
  - empty state
  - no access state
  - dashboard card/table/list item
- если используются бейджи статусов, сделай единую систему статусов
- если используются role-based differences, явно покажи их в отдельных фреймах

Что нужно выдать:
1. Обновленные employee screens
2. Новый объединенный экран Profile
3. Новый экран Dashboard для Admin
4. Вариант нижней навигации для Employee
5. Вариант нижней навигации для Admin
6. Экран No Access для попытки входа в Dashboard без прав
7. Набор ключевых состояний и edge cases
8. Компонентную структуру, пригодную для передачи в разработку

Главный принцип:
Не менять брендовый дизайн, а сделать текущий продукт визуально зрелым, логически цельным и готовым к реальной разработке mobile app + role-based dashboard.
