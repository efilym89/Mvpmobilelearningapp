Доработай текущий Figma Make проект MVP Mobile Learning App. Не делай новый редизайн и не меняй согласованный брендовый стиль. Сохрани текущую визуальную систему, композицию, цвета, типографику, скругления, карточки, иконки, отступы и общий характер интерфейса. Работай только на уровне UX, структуры экранов, ролевой логики и продуктовой целостности.

Важно:
- не ломай существующий брендовый стиль;
- не делай “другой дизайн”;
- доработай текущий проект до состояния, в котором по нему удобно реализовывать frontend, backend, API, роли, права доступа и продуктовую логику;
- пользовательский интерфейс оставь на русском языке;
- названия экранов, компонентов и слоев делай внятными и системными.

Контекст:
Это внутреннее мобильное LMS-приложение для сотрудников.
Есть 2 роли:
1. Employee
- обычный пользователь
- проходит курсы, уроки, квизы, тесты
- не имеет доступа к админке

2. Admin
- пользователь с выданными правами
- кроме обычного пользовательского сценария имеет доступ к дополнительной вкладке и странице “Дашборд”

Текущее состояние проекта нужно исправить:
- Dashboard уже существует как отдельный экран, но продуктовая структура еще не доведена;
- вкладка “Прогресс” все еще существует отдельно, а должна быть объединена с “Профиль”;
- нижняя навигация должна зависеть от роли;
- нужно сделать систему состояний и ролевой логики визуально понятной и пригодной для дальнейшей разработки.

Что нужно сделать:

1. Нижняя навигация
Сделай два явных варианта нижней навигации:
- Employee nav:
  - Главная
  - Курсы
  - Профиль
- Admin nav:
  - Главная
  - Курсы
  - Профиль
  - Дашборд

Важно:
- вкладка “Прогресс” должна исчезнуть как отдельная вкладка;
- визуально обе навигации должны быть в одном стиле;
- отличие должно быть только в наличии вкладки “Дашборд” у Admin;
- покажи оба варианта явно в макете.

2. Объединение “Прогресс” и “Профиль”
Полностью переработай экран “Профиль” так, чтобы он стал единым экраном пользователя и включал:
- блок профиля пользователя;
- должность / роль;
- статистику обучения;
- текущий прогресс;
- завершенные курсы;
- средний балл;
- часы обучения;
- достижения;
- сертификаты;
- уровень / рост / прогресс пользователя;
- при необходимости быстрые действия: настройки, поддержка, выход.

Важно:
- это не должна быть просто склейка двух старых экранов;
- экран должен выглядеть цельно, красиво и логично;
- это должен быть единый “личный кабинет” сотрудника.

3. Dashboard
Оставь страницу “Дашборд”, но доведи ее до понятного admin-MVP экрана в том же брендовом стиле.

Dashboard должен покрывать базовые сценарии администрирования:
- курсы;
- сотрудники;
- назначения;
- тесты и квизы;
- отчеты / статистика.

Важно:
- не делать тяжелую enterprise-панель;
- сделать легкий, чистый, мобильный dashboard;
- сохранить стиль приложения, чтобы dashboard выглядел частью того же продукта.

4. Access control / No Access
Сделай отдельный, полноценный экран или состояние:
- “Нет доступа к Дашборду” / “Доступ запрещен”

Этот экран нужен для сценария:
- пользователь без admin-прав каким-то образом открывает Dashboard

Экран должен:
- быть оформлен в текущем стиле продукта;
- объяснять, что доступ ограничен;
- предлагать понятное действие:
  - вернуться на главную
  - обратиться в поддержку / запросить доступ

Важно:
- это должно быть не просто сообщение, а полноценный продуманный экран состояния.

5. Role-based structure
Покажи в макете структуру приложения отдельно для:
- Employee flow
- Admin flow

Нужно явно отразить:
- какие экраны доступны обычному пользователю;
- какие экраны доступны администратору;
- чем отличается навигация;
- чем отличается доступ.

6. Каталог курсов
Сохрани уже улучшенную идею каталога и доведи ее до целостного состояния.
В каталоге должны быть понятны:
- обязательный курс;
- рекомендованный курс;
- назначенный курс;
- статус “в процессе”;
- дедлайн;
- прогресс.

Если каких-то состояний не хватает визуально, добавь их, но без перегруза.

7. Структура данных и состояний в дизайне
Сделай так, чтобы из макета было визуально понятно, что существуют и поддерживаются следующие сущности/состояния:
- роль пользователя;
- is admin / no admin access;
- assigned course;
- mandatory course;
- recommended course;
- deadline;
- course status;
- lesson status;
- quiz status;
- final test status;
- certificate availability.

Не нужно писать backend-документацию в макете, но дизайн должен поддерживать эту модель, а не конфликтовать с ней.

8. Подготовка к handoff
Подготовь проект так, чтобы после него было удобно проектировать разработку.
Нужно явно и аккуратно показать:
- Employee navigation
- Admin navigation
- Unified Profile
- Dashboard
- No Access state
- Catalog states
- Role-based differences

Что нужно выдать в результате:
1. Обновленный экран Profile как объединение старых Profile + Progress
2. Employee bottom nav на 3 вкладки
3. Admin bottom nav на 4 вкладки
4. Обновленный Dashboard
5. Экран / состояние “Нет доступа к Дашборду”
6. Явное визуальное разделение Employee flow и Admin flow
7. Сохранение всей текущей брендовой стилистики без редизайна

Главный принцип:
Не изобретай новый продукт визуально.
Приведи текущий дизайн к логически завершенному состоянию, чтобы он был согласован не только по бренду, но и по реальной продуктовой архитектуре, ролям и будущей разработке.
